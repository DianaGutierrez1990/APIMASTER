Cómo trabaja cada canal

🖥️ Desks (Desktop apps)

Ruta directa — no pasan por el API
  1. El Desk abre con su connection string local hardcoded
  2. Se autentica contra SQL Server con un SQL login propio
  3. Ejecuta TVFs/SPs directamente en la BD
  4. No hay red intermedia, no hay API que pueda caerse

Ventaja: baja latencia, no depende del API.
Limitación: solo funciona dentro de la red corporativa con acceso al puerto 1433.

🌐 Web / Móvil / Terceros

Vía APIMASTER — canal único HTTP
  1. Hace POST /api/secure/LoginStd a login.intelogix.net y guarda el JWT
  2. Envía requests a APIMASTER con Authorization: Bearer <JWT>
  3. APIMASTER valida firma HMAC, descifra la connection string del token
  4. APIMASTER conecta a la BD correcta del software (Incidents, Payroll, etc.) y ejecuta la TVF/SP
  5. Devuelve JSON al cliente

Ventaja: funciona desde internet, sin exponer SQL al exterior.
Escalado: N instancias stateless del API detrás de un Load Balancer.

¿Qué pasa si se cae algo?

Si cae el API

Los Desks siguen operando con su conexión directa a SQL. Web/móvil dejan de funcionar hasta que vuelva el API. Solución: correr 2+ instancias detrás de Load Balancer — si una muere, las otras absorben.

Si cae login.intelogix.net

No se pueden emitir tokens nuevos, pero los tokens ya emitidos siguen siendo válidos hasta su expiración (exp claim). Los Desks no se afectan.

Si cae SQL Server

Nada funciona — ni Desks ni API. Es la única dependencia dura. Mitigación: replicación + backups (responsabilidad de infra, no del API).
