Sección 01

Resumen Ejecutivo

Se completó la migración de la API de producción de la plataforma Dairy, pasando de una implementación en Python Flask a una nueva arquitectura basada en .NET 10 Web API con soporte multi-tenant, autenticación centralizada con Keycloak, y múltiples capas de seguridad.

La API anterior presentaba 7 vulnerabilidades críticas de seguridad y 7 problemas arquitectónicos que ponían en riesgo la integridad de los datos, la disponibilidad del servicio y el cumplimiento de estándares de seguridad. Todas estas vulnerabilidades han sido remediadas en la nueva implementación.

Sección 02

Análisis de Vulnerabilidades del Sistema Anterior

La API en Python Flask presentaba las siguientes vulnerabilidades de seguridad identificadas durante el análisis previo a la migración. Se clasifican según severidad.

Vulnerabilidades de Seguridad

Crítica — OWASP A03:2021 Injection
V-01: SQL Injection en todos los endpoints
Las consultas SQL se construían con f-strings de Python, concatenando directamente los parámetros del usuario en el SQL: f"SELECT * FROM MV_List('{startDate}', '{endDate}')". Un atacante podía inyectar SQL arbitrario y acceder, modificar o eliminar datos de cualquiera de las 6 bases de datos.
Crítica — OWASP A07:2021 Authentication Failures
V-02: Sin autenticación
La API no tenía ningún middleware de autenticación. Cualquier persona o bot con acceso a la URL http://168.197.51.133:9070 podía invocar cualquier endpoint sin restricción.
Crítica — OWASP A02:2021 Cryptographic Failures
V-03: Credenciales hardcodeadas en código fuente
El archivo config/database.py contenía en texto plano los usuarios, contraseñas e IPs de 6 servidores SQL Server y el archivo config/ftpConfig.py las credenciales FTP. Cualquier persona con acceso al repositorio obtenía acceso directo a todos los servidores.
Alta — OWASP A01:2021 Broken Access Control
V-04: El cliente elige a qué base de datos conectarse
El body del request incluía un campo "db": "General_Dairyi" que permitía al cliente seleccionar la base de datos destino. Un atacante podía apuntar a cualquier BD del diccionario de conexiones.
Alta — OWASP A05:2021 Security Misconfiguration
V-05: CORS abierto sin restricción de origen
La configuración CORS(app) sin parámetros permitía requests desde cualquier dominio, facilitando ataques Cross-Site Request Forgery (CSRF).
Alta — OWASP A05:2021 Security Misconfiguration
V-06: Debug activo en producción
app.run(debug=True) exponía stack traces completos con rutas de archivos, variables y código fuente ante cualquier error, dando información valiosa a un atacante.
Media — OWASP A04:2021 Insecure Design
V-07: Sin headers de seguridad HTTP
No se enviaban headers como X-Content-Type-Options, X-Frame-Options, Content-Security-Policy ni Strict-Transport-Security.

Problemas Arquitectónicos

Arquitectura
A-01: Sin validación de input
No se validaban tipos, formatos, rangos ni longitudes de los parámetros recibidos.
Arquitectura
A-02: Sin rate limiting
Un cliente podía hacer requests ilimitados, abriendo la puerta a ataques de denegación de servicio (DoS).
Arquitectura
A-03: Sin auditoría
No existía registro de qué usuario ejecutaba qué operación ni cuándo.
Arquitectura
A-04: Sin paginación
Los endpoints retornaban todos los resultados sin límite, potencialmente miles de registros en una sola respuesta.
Arquitectura
A-05: Todos los endpoints usan POST
Incluso las consultas de solo lectura usaban POST, violando los estándares REST.
Arquitectura
A-06: Sin versionado de API
Las URLs no tenían versión, imposibilitando cambios sin romper clientes existentes.
Arquitectura
A-07: Instancia única sin failover
Una sola instancia Flask ejecutándose en un servidor. Si caía, todo el servicio se perdía.
Sección 03

Arquitectura de la Nueva API (.NET 10)

La nueva API fue diseñada con una arquitectura en capas que separa responsabilidades y permite escalar cada componente de forma independiente.

Stack Tecnológico

Componente Tecnología Propósito
Framework .NET 10 Web API Framework principal, versión LTS más reciente con soporte hasta Nov 2028
Acceso a datos Dapper 2.1 Micro-ORM para ejecutar funciones y procedimientos de SQL Server
Autenticación JWT Bearer + Keycloak Servidor de identidad centralizado que emite tokens de acceso
Validación FluentValidation 11.3 Validación declarativa de todos los datos de entrada
SQL Server Microsoft.Data.SqlClient 7.0 Driver moderno para SQL Server con encriptación de conexión
Logging Serilog 10.0 Sistema de registro de eventos con contexto de empresa y usuario
Documentación Swagger / OpenAPI Documentación interactiva y explorable de la API
Multi-Tenancy Entity Framework Core 10 Interceptor de conexión para aislamiento de datos por empresa

¿Qué es cada tecnología y por qué se eligió?

.NET 10 Web API

.NET 10 es la versión más reciente del framework de Microsoft para construir aplicaciones web de alto rendimiento. Es una versión LTS (Long-Term Support), lo que significa que Microsoft garantiza actualizaciones de seguridad y parches hasta noviembre de 2028. Se eligió esta versión para mantener el proyecto en la tecnología más actual disponible y maximizar el tiempo de vida del soporte oficial.

Dapper — Acceso a Datos

Dapper es un micro-ORM (Object-Relational Mapper) de alto rendimiento desarrollado por el equipo de Stack Overflow. Se eligió Dapper en lugar de un ORM completo porque las 6 bases de datos SQL Server ya existían con todas sus funciones (Table-Valued Functions) y procedimientos almacenados (Stored Procedures) creados y en uso. No se necesitaba crear tablas ni migraciones — solo ejecutar las funciones existentes de forma segura.

Dapper actúa como puente entre el código C# y SQL Server: envía las consultas con parámetros seguros (eliminando el riesgo de SQL Injection que existía en Python) y convierte automáticamente los resultados de SQL en objetos C# que la API retorna como JSON. Es hasta 10x más rápido que un ORM completo para operaciones de solo lectura como las de este proyecto.

Keycloak + JWT — Autenticación

Keycloak es un servidor de identidad open-source desarrollado por Red Hat (la misma empresa detrás de Linux Red Hat Enterprise). Funciona como el “guardia de seguridad” centralizado de toda la plataforma: es el único sistema que verifica credenciales de usuarios y emite permisos de acceso.

JWT (JSON Web Token) es el formato estándar de la industria para tokens de acceso. Cuando un usuario se autentica en Keycloak, recibe un token JWT firmado digitalmente que contiene: su identidad, a qué empresa pertenece (tenant_id), y qué permisos tiene (roles). Este token se envía en cada solicitud a la API, que lo valida automáticamente sin necesidad de consultar a Keycloak en cada petición — logrando máxima seguridad con mínimo impacto en rendimiento.

Se eligió Keycloak por ser gratuito, on-premise (corre en nuestros propios servidores, sin depender de servicios en la nube), soporta Single Sign-On (SSO), protección contra fuerza bruta, y es usado por organizaciones como Bosch, Lufthansa y el Gobierno de Reino Unido.

Serilog — Logging Estructurado

Serilog es un sistema de registro de eventos (logging) que captura cada operación de la API con información contextual rica: qué usuario hizo la operación, de qué empresa es, cuánto tiempo tardó, y si fue exitosa o falló. A diferencia de los logs básicos de texto (como los que tenía la API en Python), Serilog genera logs estructurados que se pueden consultar, filtrar y analizar — por ejemplo: “mostrar todas las operaciones fallidas de la empresa X en los últimos 7 días”.

Esto es esencial para auditoría (saber quién hizo qué y cuándo), diagnóstico de problemas (identificar rápidamente la causa de errores), y monitoreo de rendimiento (detectar endpoints lentos o patrones de uso inusuales).

Entity Framework Core 10 — Multi-Tenancy (50+ empresas)

Entity Framework Core se utiliza específicamente para implementar el aislamiento multi-tenant de la plataforma. Multi-tenancy significa que una sola instancia de la API atiende a más de 50 empresas simultáneamente, pero cada empresa solo puede ver y modificar sus propios datos — como si cada una tuviera su propio sistema privado.

El mecanismo funciona así: cada vez que la API abre una conexión a la base de datos, el interceptor de EF Core ejecuta automáticamente un comando SQL (sp_set_session_context) que le dice a SQL Server: “esta conexión pertenece a la empresa X”. SQL Server entonces aplica Row-Level Security (RLS), un mecanismo nativo del motor de base de datos que filtra automáticamente las filas para que solo se devuelvan datos de esa empresa. Esto garantiza el aislamiento incluso si hubiera un error en el código de la API.

Sección 04

Seguridad — Autenticación con Keycloak

Keycloak es un Identity Provider open-source (Red Hat) que centraliza la autenticación de todos los usuarios y aplicaciones. Reemplaza la ausencia total de autenticación que tenía la API anterior.

Flujo de Autenticación

1. El usuario abre la app móvil o web | v 2. La app redirige al login de Keycloak Usuario ingresa credenciales | v 3. Keycloak valida credenciales y genera un TOKEN JWT El token contiene: - sub: ID del usuario - tenant_id: UUID de la empresa - tenant_name: Nombre de la empresa - roles: [admin, writer, reader] - exp: Expiración (15 minutos) | v 4. La app envía el token en cada request a la API Header: Authorization: Bearer eyJhbG... | v 5. La API valida el token LOCALMENTE - Verifica firma con llave pública de Keycloak (JWKS) - Verifica que no esté expirado - Verifica que el audience sea "api-principal" - Extrae tenant_id y roles de los claims | v 6. Request autorizado → se ejecuta el endpoint

Configuración del Realm

El realm "plataforma" en Keycloak está configurado con:

Claims Personalizados en el Token

Keycloak inyecta claims personalizados en cada token JWT mediante Protocol Mappers:

ClaimTipoEjemploUso en API
tenant_id String (UUID) 00000000-0000-0000-0000-000000000001 Identificar empresa, filtrar datos, SESSION_CONTEXT
tenant_name String Empresa Demo Logging y auditoría
roles Array ["admin", "writer"] Políticas de autorización
sub String (UUID) UUID del usuario Identificar usuario en auditoría
Importante: La API nunca genera tokens. Keycloak es el único emisor. La API solo valida la firma del token usando las llaves públicas descargadas automáticamente desde el endpoint JWKS de Keycloak. Esto significa que la API no necesita conexión a Keycloak en cada request — solo al arrancar.
Sección 05

Seguridad — Multi-Tenancy y Aislamiento de Datos

La arquitectura multi-tenant permite que una sola instancia de la API atienda a múltiples empresas (tenants) con aislamiento total de datos. Cada empresa solo puede ver y modificar sus propios datos.

Capas de Aislamiento

CAPA 1: Token JWT Keycloak emite tokens con tenant_id específico por usuario Un usuario de Empresa A nunca recibe un token con tenant de Empresa B CAPA 2: Tenant Middleware Extrae tenant_id del token JWT en cada request Rechaza con 403 Forbidden si no tiene tenant_id válido Lo inyecta como servicio scoped para toda la pipeline CAPA 3: SQL Server SESSION_CONTEXT El TenantDbInterceptor ejecuta en CADA conexión a BD: EXEC sp_set_session_context @key=N'TenantId', @value=@tid, @read_only=1; El @read_only=1 impide que código posterior lo modifique CAPA 4: Row-Level Security (RLS) SQL Server filtra automáticamente las filas según el TenantId en SESSION_CONTEXT Incluso si hay un bug en la API, la BD no devuelve datos de otro tenant

Políticas de Autorización

PolíticaRequisitosAplicada en
TenantReader Autenticado + claim tenant_id Todos los endpoints GET (lectura)
TenantWriter Autenticado + tenant_id + rol writer o admin Endpoints POST/PUT (escritura)
TenantAdmin Autenticado + tenant_id + rol admin Operaciones administrativas

Rate Limiting por Tenant

Se implementa una ventana deslizante (sliding window) por tenant para prevenir abuso y ataques DoS.

ParámetroValor

Security Headers

Cada respuesta de la API incluye headers de seguridad HTTP:

HeaderValorProtección
X-Content-Type-OptionsnosniffPreviene MIME sniffing
X-Frame-OptionsDENYPreviene clickjacking
Content-Security-Policydefault-src 'self'Restringe fuentes de contenido
Strict-Transport-Securitymax-age=31536000; includeSubDomainsFuerza HTTPS

Server(eliminado)Oculta software del servidor

Manejo Global de Errores

El ErrorHandlingMiddleware captura todas las excepciones y retorna respuestas en formato RFC 7807 (Problem Details). Nunca se exponen stack traces ni información interna en producción.

// Ejemplo de respuesta de error (producción) { "type": "https://api.tuempresa.com/errors/validation", "title": "Validation error", "status": 400, "detail": "One or more fields failed validation.", "requestId": "0HN8ABC123", "errors": { "startDate": ["startDate is required"], "endDate": ["endDate must be after startDate"] } }

Auditoría

El AuditMiddleware registra cada operación con:

  • Método HTTP y ruta
  • Código de respuesta
  • ID del usuario (del claim sub)
  • ID del tenant
  • Duración del request
  • Operaciones de escritura (POST/PUT/DELETE) se logean con nivel AUDIT

Consultas SQL Parametrizadas

Remediación de SQL Injection
Todas las consultas usan parámetros con Dapper
Se reemplazaron las f-strings de Python por queries parametrizadas. Los valores del usuario nunca se concatenan en el SQL.
// ANTES (Python) - VULNERABLE: f"SELECT * FROM MV_List('{startDate}', '{endDate}')" // AHORA (.NET) - SEGURO: conn.QueryAsync<T>( "SELECT * FROM MV_List_Milk_Customers(@startDate, @endDate, @locationId)", new { startDate, endDate, locationId });

CORS Restringido

Remediación de CORS abierto
Solo orígenes autorizados pueden hacer requests
Producción: solo https://app.tuempresa.com. Desarrollo: localhost:3000 y localhost:5173. Configurable via appsettings.json.

Sección 09

Pipeline de Middleware

Cada request HTTP pasa por la siguiente cadena de middleware en orden estricto. El orden es crítico: la autenticación debe ocurrir antes de la autorización, y el tenant debe resolverse después de autenticar.

1 Security Headers Agrega headers de seguridad HTTP a la respuesta
↓
2 Error Handling Captura excepciones → RFC 7807 Problem Details
↓
3 HTTPS Redirect Redirige HTTP a HTTPS
↓
4 CORS Valida orígenes permitidos
↓
5 Rate Limiter 200 req/min por tenant (ventana deslizante)
↓
6 Authentication Valida token JWT contra Keycloak JWKS
↓
7 Authorization Verifica políticas: TenantReader / TenantWriter / TenantAdmin
↓
8 Tenant Middleware Extrae tenant_id del JWT → TenantContext scoped
↓
9 Audit Middleware Registra operación con usuario, tenant, duración
↓
10 Controller + Service Ejecuta la lógica de negocio

Validación de Datos

Todos los inputs del usuario se validan antes de llegar a la lógica de negocio usando FluentValidation. Se implementaron 22+ validadores que cubren todos los endpoints de la API.

Ejemplos de Reglas de Validación

ValidadorReglas
MilkCustomersRequest StartDate requerido, EndDate > StartDate, LocationId > 0
MilkLoadsSearch Al menos Ticket o Manifest debe tener valor
UpsertMilkLoad IdCustomer > 0, IdUser > 0, Pesos >= 0, ImageType en lista válida, Imagen ≤ 10MB
PaginationParams Page ≥ 1, PageSize entre 1 y 100 (default 20)
ChangeStatus QuestionId requerido, Status en valores válidos
Auto-validación: FluentValidation está configurado con AddFluentValidationAutoValidation(). Cualquier request que no pase la validación retorna automáticamente un HTTP 400 con los errores detallados en formato RFC 7807, sin llegar al controller.

Tests de Integración — Ejemplo

Los tests de integración usan CustomWebApplicationFactory que levanta la API completa en memoria con un TestAuthHandler que simula tokens JWT con claims de tenant y roles, permitiendo probar todo el pipeline sin depender de Keycloak.
