API Única vs API por Software

Ventajas, desventajas y cuándo conviene cada arquitectura

Proyecto: APIMASTER — Ecosistema Intelogix
Autor: Diana Gutiérrez
Empresa: Intelogix
Fecha: 16 de abril, 2026
Documento de decisión arquitectónica

El dilema

Intelogix tiene varios productos (Softwares): Incidents, Scalei, CreditCharges, Diesel, StaffManager, Alimentos, Payroll, Approvals, etc. Cada uno consume su propia base de datos. La pregunta es:

¿Un solo API que atiende a todos los softwares, o un API por cada software?

Este documento compara las dos opciones objetivamente para tomar la decisión.

Opción A — API Única

Un solo API que atiende a todos los softwares
Web / Móvil / Terceros │ ▼ ┌──────────────────┐ │ APIMASTER │ ← un solo proyecto │ │ │ /api/v1/staff/ │ │ /api/v1/credit/ │ │ /api/v1/diesel/ │ │ /api/v1/scalei/ │ └────────┬─────────┘ │ ┌──────┼──────┬──────┐ ▼ ▼ ▼ ▼ StaffDB CrdDB DieselDB ScaleDB
Ventajas
  • Simple de desarrollar — un solo repo, un solo proyecto, un solo deploy
  • Infraestructura mínima — una sola máquina o contenedor basta para arrancar
  • Autenticación centralizada — el JWT se valida una sola vez, no en N APIs
  • Debugging sencillo — un solo log, un solo stacktrace
  • Compartir código — validaciones, servicios comunes, DTOs se reutilizan sin duplicar
  • Menor costo inicial — no pagas servidores separados por módulo
  • Deploy atómico — todo sube junto, no hay "versiones desfasadas" entre servicios
Desventajas
  • Punto único de falla — si cae, TODOS los softwares se quedan sin API
  • Escalado global — si Diesel tiene mucho tráfico, tienes que escalar TODO el API, no solo Diesel
  • Deploy arriesgado — un bug en Staff puede bloquear el deploy de una mejora en Credit
  • Equipo en conflicto — varios devs editando el mismo proyecto → merges dolorosos
  • Release conjunto — si Credit necesita publicar hoy, tiene que esperar a que Staff esté listo
  • Crece feo con el tiempo — si no disciplinas la estructura, se vuelve un "big ball of mud"
  • Tecnología unificada — todos los módulos deben ser .NET 10 (no puedes hacer uno en Python si hace falta)
📌 Cuándo conviene: pocos módulos (<10), equipo pequeño (1-5 devs), primeros 1-2 años del proyecto, quieres mover rápido sin invertir en infra.

Opción B — API por Software

Un API independiente para cada producto (Incidents, Scalei, CreditCharges…)
Web / Móvil / Terceros │ ▼ ┌──────────────────┐ │ API Gateway │ ← enruta por prefijo └────────┬─────────┘ │ ┌────┬──┴──┬────┬────┐ ▼ ▼ ▼ ▼ ▼ Staff Credit Diesel Scalei Incidents API API API API API │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ StaffDB CrdDB DieselDB ScaleDB IncDB
Ventajas
  • Aislamiento de fallas — si cae Diesel API, Staff y Credit siguen funcionando
  • Escalado selectivo — Diesel con 10× tráfico → 10 instancias de Diesel; Staff con 1× → 1 instancia
  • Deploys independientes — cada software se publica cuando esté listo, sin esperar a otros
  • Equipos paralelos — un dev en Credit, otro en Staff, otro en Diesel, sin chocar en el mismo repo
  • Tecnología por módulo — Staff puede ser .NET 10, Alimentos puede ser Python, Scalei puede ser Go
  • Responsabilidad clara — el equipo que mantiene Staff es dueño del repo, deploy y uptime de Staff
  • Límite de blast radius — un bug grave solo tira a un software, no a todos
  • Preparado para crecer — este patrón escala bien a 20, 50 o 100 módulos
Desventajas
  • Infraestructura compleja — necesitas N servidores/containers + un API Gateway + orquestador (Kubernetes, Docker Swarm)
  • Más repos y pipelines CI/CD — cada API es un proyecto con su propio build, tests, deploy
  • Latencia extra — cada salto HTTP suma milisegundos (request pasa por Gateway → Bridge API)
  • Debugging distribuido — un error puede atravesar 3 servicios → necesitas tracing (OpenTelemetry, Jaeger)
  • Duplicación de código — validadores, middlewares, auth handlers se repiten en N proyectos (o sacas librerías compartidas)
  • Coordinación entre módulos — si Staff cambia un contrato, Credit puede romperse → necesitas contratos/versionado
  • Costo operativo mayor — más máquinas, más logs, más monitoreo, más DevOps
  • Overkill al principio — montar toda la infra sin tener tráfico real es desperdicio
📌 Cuándo conviene: muchos módulos (10+), equipo grande (>8 devs), cada software tiene patrones de tráfico muy distintos, presupuesto para DevOps, requieres 99.9%+ de uptime por software.

Resumen del conteo: API Única gana en 7 criterios (simplicidad, costo, velocidad, consistencia). API por Software gana en 6 criterios (resiliencia, escalado, independencia, flexibilidad). No hay un ganador absoluto — depende del momento del proyecto.

Recomendación para Intelogix

🎯 Evolución por fases — lo mejor de los dos mundos

No hay que elegir de una vez. La estrategia es empezar monolítico pero bien estructurado y migrar a API por software solo cuando el tráfico y el equipo lo justifiquen.

Fase 1 — Hoy (0 a 12 meses): API Única bien modularizada

Un solo proyecto APIMASTER, pero con carpetas por software (Modules/Staff/, Modules/Credit/, Modules/Diesel/). Cada carpeta tiene sus propios Controllers, Services y Entities. Internamente ya opera como si fueran APIs separadas, pero sin el costo de infraestructura.

Fase 2 — Cuando crezca (12 a 24 meses): Identificar módulos que deben separarse

Si un software tiene mucho más tráfico, cambia más seguido, o requiere uptime distinto, se saca a su propio proyecto. El resto sigue en APIMASTER. Se agrega un API Gateway (YARP, Kong) al frente.

Fase 3 — A gran escala (24+ meses): API por software consolidado

Cuando el negocio crezca, cada software vive independiente con su propio equipo, repo, deploy y SLA. APIMASTER se convierte en el Gateway o queda como uno más de los bridges.

💡 La clave: no pagar hoy la complejidad que solo necesitarás en 2 años, pero estructurar el código para que la migración sea trivial cuando llegue el momento.
