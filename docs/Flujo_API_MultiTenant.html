1Modelo actual de la empresa

Cada software/producto del Desk maneja múltiples empresas. Todas las empresas viven en una misma BD, diferenciadas por una columna ID_Business (uniqueidentifier / GUID).

Servidor SQL (35.223.136.179) │ ├── CreditCharges → Tarjetas, transacciones, gastos ├── Diesel / Diesel_v3 → Combustible ├── Alimentos_Integrados → Alimentación ├── InventoryAsset → Vehículos / activos ├── Payroll → Nómina ├── StaffManagerUtilities → Contratos, vacaciones, warnings ├── Employees_General_Info → Catálogo maestro de empleados └── Approval_Prod → Aprobaciones
Catálogo central de empresas: CreditCharges.web.vw_business
Columnas: id_business, short_name, business.
Actualmente hay más de 40 empresas registradas (Intepizzaco, Airman, Scoria, Veribest, Lipan Cattle Feeder, In4Life Farms, Richies Towing, etc.).

Cada tabla operativa (empleados, transacciones, vehículos, combustible, alimentos) lleva ID_Business para saber a qué empresa pertenece el registro.

2Arquitectura Thin API — cómo opera

Principio de diseño: el API se comporta como los Desks. No administra permisos ni lógica de tenant — eso ya lo resuelve StaffManager en el frontend y la BD mediante la columna ID_Business.

El API .NET 10 NO cambia la base de datos, NO implementa multi-tenancy, NO decide qué puede ver el usuario. Solo:

  1. Recibe el token JWT del usuario (emitido por Keycloak) para autenticar que el llamador es válido.
  2. Recibe los parámetros del request (incluido id_business) y valida el formato con FluentValidation.
  3. Llama al TVF/SP correcto pasando los parámetros tal cual los manda el frontend.
  4. Devuelve el JSON con lo que SQL Server retorna, sin filtrado extra.

El permiso de lectura/escritura por empresa ya está resuelto: el Desk solo mostró al usuario las empresas a las que tiene acceso, y el frontend solo manda requests con los id_business autorizados.

4Estructura del API por módulo

Cada módulo del Desk (StaffManager, CreditCharges, Diesel, Alimentos, Vehículos) tiene su propio conjunto de endpoints, pero todos comparten el mismo patrón de filtrado por ID_Business.

5Flujo de desarrollo por módulo

Para cada módulo se ejecutan los mismos 5 pasos:

  1. Introspección de BD. Revisar columnas reales de las TVFs/SPs y tablas para saber qué parámetros acepta cada una.
  2. Entidades C#. Crear clases que reflejen 1:1 las columnas reales (sin inventar nombres).
  3. Request DTOs + FluentValidation. Validar formato del input (que id_business sea GUID, que las fechas sean válidas, etc.).
  4. Service con Dapper. Llamar TVF/SP con id_business y demás parámetros que vienen del request — tal cual los manda el frontend.
  5. Controller con [Authorize]. Ruta REST, verbos HTTP correctos (GET/POST/PUT/DELETE), respuestas con código HTTP adecuado.

7Beneficios para el negocio

Una sola API Todos los productos del Desk (StaffManager, CreditCharges, Diesel, etc.) consumen api.intelogix.mx — un solo punto de mantenimiento.
Seguridad por empresa Un usuario de la empresa A nunca puede ver datos de la empresa B, aunque manipule el frontend — el filtrado ocurre en el servidor.
Escalable Agregar una empresa nueva = insertar un row en web.vw_business, sin tocar una sola línea de código.
Auditable Cada request queda registrado con usuario + ID_Business + acción + timestamp.
Migración gradual Los módulos se mueven de Python a .NET uno por uno, sin romper lo que ya funciona.
Alineado al modelo existente No se rediseña la BD ni se pide al equipo de datos cambios estructurales — se respeta ID_Business.
